2. New Problems Introduced by v2 Fixes

Transaction-Per-Request Overhead

The SET LOCAL fix wraps every admin API request in a database transaction (BEGIN...COMMIT). For read-only GET requests (listing properties, viewing calls), this adds unnecessary transactional overhead. Consider using SET LOCAL only for write operations, or using a read-only connection without RLS for non-tenant-scoped queries (like tool call endpoints that look up agents by retell_agent_id).

Tool Call Endpoints Still Bypass RLS

The tool call endpoints (search-knowledge, check-availability) query agents and knowledge_chunks directly via pool.query() without tenant context middleware. This is by design (Retell calls them mid-conversation, no auth header), but it means these endpoints bypass RLS entirely. They filter by retell_agent_id to find the tenant, then query with explicit WHERE tenant_id = $1. This is functionally correct but creates two different data access patterns — RLS for admin routes, manual filtering for tool routes. If someone adds a new tool endpoint and forgets the manual tenant filter, there's no safety net.

Consent Type Inconsistency

v2 keeps the consent_type as 'informational' | 'marketing' in the schema while simultaneously stating that the attorney needs to review all consent language. If the attorney determines (as expected) that all AI outbound requires written consent, the informational consent type becomes misleading or useless. The schema should probably use 'express_written' | 'prior_express_written' or similar legally accurate terminology, or at minimum note this as pending attorney review.

3. Issues the v1 Council Missed

No Database Connection Cleanup on Tool Call Endpoints

The search-knowledge and check-availability handlers use pool.query() directly, which auto-releases connections. But the check-availability handler makes an external HTTP call (fetch to the tenant's API) after acquiring its database results. If that fetch hangs for 8 seconds (its timeout), the database connection from pool.query is already released — this is fine. However, if someone refactors to use pool.connect() for these endpoints (e.g., to add caching in a transaction), the connection would be held for the full 8-second external call, potentially exhausting the pool under load.

No Idempotency on Webhook Processing

The call_ended webhook handler inserts a usage_record and queues a transcript analysis job. If Retell retries the webhook (which it does — "3 retries" per the verified specs), these operations will execute multiple times. The usage record will be duplicated (inflating billing), and multiple transcript analyses will run for the same call. The handler needs idempotency — check if a usage record already exists for this retell_call_id before inserting.

No Graceful Shutdown for Workers

The worker service processes BullMQ jobs, but there's no mention of graceful shutdown handling. If Railway restarts the worker service (deployment, scaling, crash), any in-progress transcript analysis (which calls Claude API and may take 10-30 seconds) will be interrupted. BullMQ has built-in support for graceful shutdown via worker.close(), which waits for in-progress jobs to complete. Without this, jobs will be marked as failed and retried, potentially causing duplicate processing.

Admin API Has No Input Validation

The CRUD routes accept req.body directly without Zod validation (despite Zod being in package.json). The properties POST route passes name, slug, metro_area, address, timezone, settings directly to SQL. While parameterized queries prevent SQL injection, there's no validation that slug is URL-safe, timezone is a valid IANA timezone, or settings is a valid JSON structure. This will cause subtle bugs (invalid timezones breaking the compliance gate, non-URL-safe slugs breaking widget embed codes).

`@aws-sdk/s3-request-presigner` Not in package.json

The storage service imports getSignedUrl from @aws-sdk/s3-request-presigner, but this package is not listed in the package.json dependencies. The install will fail when this service is first used.

No Health Check for Worker Service

The API server has a /health endpoint. The worker service has no equivalent. Railway needs a way to know if the worker is alive. BullMQ workers can expose a simple HTTP health endpoint that reports queue connection status and active job count.

OpenAI API Key Environment Variable Mismatch

The knowledge service references process.env.OPENAI_API_KEY, but the .env.example does not include OPENAI_API_KEY. It only has ANTHROPIC_API_KEY. The embeddings use OpenAI's text-embedding-3-small, so this key is required for Phase 1, not optional.

QUESTION 2: Did any v2 fix introduce new problems?

2a. Transaction-per-request overhead (from RLS fix)

The SET LOCAL fix wraps every HTTP request in a database transaction. For read-only GET endpoints (knowledge search, call listing, analytics), this means:

  • Every GET acquires a connection, opens a transaction, sets tenant context, queries, commits, releases
  • Under load, the connection pool saturates faster because connections are held for the entire request lifecycle (including response serialization)
  • Read-heavy dashboards (analytics, call history) hit this disproportionately

Impact: Medium. The correct fix would be to differentiate: use SET LOCAL within transactions for write operations, and use a simpler pattern (advisory lock, or just application-level filtering with WHERE tenant_id = $1) for read-only GET routes. The current approach trades correctness for performance — correctness wins, but the doc should acknowledge the pool sizing implications.

2b. Tool call endpoints bypass RLS (two data access patterns)

The tenant-context middleware (line 578-603) sets app.current_tenant_id for RLS. But the tool call endpoints (/api/tools/search-knowledge, /api/tools/check-availability) are called by Retell webhooks, not by authenticated dashboard users. The Retell webhook includes the agent_id or call_id — the middleware needs to resolve tenant from the agent/call, not from Clerk auth.

v2 doesn't address how tool call endpoints participate in the tenant context middleware. If they use extractTenantFromAuth(req) and the webhook request has no Clerk session, RLS filters break (empty current_tenant_id = no rows returned, or worse, fallback behavior that returns all rows).

Impact: High. This is a functional correctness bug. The webhook endpoints need their own tenant resolution path (look up agent → get tenant_id → set context).
