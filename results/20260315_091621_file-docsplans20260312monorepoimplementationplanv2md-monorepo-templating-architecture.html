Monorepo Architecture Review — Brutally Honest Assessment

1. ARCHITECTURE VALIDATION

Verdict: Right call, with caveats.

Turborepo + separate apps + shared packages is the correct architecture for this scenario. The evidence confirms it:

  • LA has 137 top-level components, Lofts has 135. The diff shows only ~15 components differ between them. That's ~89% identical at the component level.
  • Both repos maintain parallel copies of email.ts, rate-limit.ts, auth-client.ts, analytics.ts, seo.ts — textbook extraction candidates.
  • Both already use Tailwind v4 with identical @theme inline blocks, just different color values. The token generation approach is natural.

What I'd do differently: Nothing fundamental. The hybrid Turborepo approach (not multi-tenant, not npm packages) is correct for this exact situation. Multi-tenant single-app would be a nightmare for SEO isolation. Published npm packages would add version management overhead for a team of ~1-2 people.

One concern: The plan says "packages consumed as source" (no dist/ build step, Turbopack compiles). This is elegant for DX but creates a hard coupling to Turbopack. If you ever need to use a package outside Next.js (a CLI tool, an Edge Function, a test runner without Turbopack), you'll need to retrofit a build step. For now this is fine — just be aware of the ceiling.

2. SEO AND E-E-A-T ASSESSMENT

Verdict: Strong, but with one structural gap.

The 3-tier composition system is well-designed. The build-time validator that fails on identical layout + sections for indexed routes is the right guardrail. The per-domain Analytics/GSC/PostHog/Sentry split is correct (CoStar model).

Remaining risks:

  • domain_visibility is opt-out, not opt-in. The default is ARRAY['luxuryapartments','lofts','d3','d4'] — meaning every listing appears on every domain by default. This is correct for coverage but means two domains could serve near-identical search/city/neighborhood pages with 95% overlapping listings. The composition manifest catches layout differences but NOT data overlap. If LA's /cities/boston and Lofts' /cities/boston both serve the same 200 apartments with only different CSS, Google sees them as thin doorway variants. Mitigation needed: Each domain should have meaningfully different listing curation, not just different paint. The plan doesn't address content differentiation at the data layer.
  • Per-domain blog tables are correct (la_blog_posts, ll_blog_posts). But the plan doesn't prevent copy-paste of actual blog content between domains. See question 9.
  • Tier 2 "configurable" pages are the highest risk surface. The validator checks layout + sections identity, but two domains could have different section names mapping to identical content. E.g., sections: ['hero', 'stats', 'neighborhoods'] vs sections: ['hero-banner', 'market-data', 'areas'] — different labels, identical output.

What's adequate: Separate GSC properties, separate GA4, footer cross-links (transparent portfolio), isProductionDomain per app, separate Vercel projects. The robots/sitemap/noindex layer is already battle-tested on LA.

3. TOP 3 RISKS

Risk 1: Phase 2 UI extraction breaks things silently

Moving 60+ components across 4 waves, with import path rewrites across 900+ files, is the highest-risk operation in the entire plan. The "build gate" (does it compile?) catches type errors but NOT visual regressions. You have zero tests on LA and 209 on Lofts. A component that renders differently after extraction — wrong CSS var resolution, missing Tailwind @source path, broken relative import — won't be caught by the build.

Missing mitigation: Visual regression testing. The plan mentions "future: Chromatic or Percy" in Phase 4 — but Phase 2 is where you need it. At minimum, screenshot 20 critical pages before and after each wave and diff them.

Risk 2: Lofts data migration (Phase 0) is higher-effort than scoped

Lofts has 185,000 lines of static JSON data that needs to migrate to Supabase. The plan says "write a seed script using idempotent upserts" — but Lofts' current apartments-server.ts is a hybrid static+MLS layer with a 60-line transformRow function mapping between different schemas. The static data shape and Supabase schema may not be 1:1. This is a data mapping exercise, not just a bulk insert.

Missing mitigation: A schema comparison step between Lofts' static data shape and LA's Supabase schema before writing the seed script. The plan assumes they match.

Risk 3: Single-person bus factor with 4 Vercel projects, 1 Supabase DB, shared packages

A bad migration to the shared Supabase DB takes down 4 sites simultaneously. A broken shared package change (even with pnpm build:all) could ship if the CI gate has a timing window. The plan says "rollback: revert the commit" — but with 4 Vercel projects auto-deploying on main push, you need coordinated rollback across all 4.

Missing mitigation: A deployment strategy that doesn't auto-deploy all 4 apps simultaneously. Stagger deploys. Have a "canary app" (probably LA since it's the most mature) that deploys first, with a 30-minute soak time before others follow.

4. SEQUENCING

Verdict: Phase 0 first is the right call.

Evidence supports this strongly:

  • Lofts has 49 "use client" pages that need conversion (their TRACKING.md lists it as "biggest perf win")
  • Lofts is still on 185K lines of static JSON — it can't share the packages/data/ layer until it's on Supabase
  • Lofts has 209 tests; LA has zero. Extracting into shared packages without tests is dangerous.

Launching Lofts standalone first means:

  • Lofts gets to production faster (revenue)
  • The data migration is validated on a real deployment
  • The 209 tests provide a regression safety net when packages are later extracted
  • Phase 3 (porting Lofts into monorepo) is simpler because Lofts is already on Supabase

One adjustment: Phase 1 (scaffold monorepo) should NOT start "in parallel with Phase 0" as the dependency diagram suggests. Start it after Phase 0.1B (data parity verified). Otherwise you're scaffolding a monorepo while the data layer of one app is in flux — and you'll end up extracting into packages/data/ twice (once for static, once for Supabase).

5. MISSING CONCERNS

Developer experience: The plan is solid here — source consumption (no dist/), Turbopack hot reload across packages, ESLint enforcement. What's missing: how do you develop locally against a shared Supabase DB? If LA and Lofts both point at the same Supabase project during development, test data from one pollutes the other. The plan should specify: one Supabase project per environment (dev/staging/prod), or Supabase branching for isolated dev.

Content workflow: Biggest gap. Who writes blog content for each domain? How do they know what's unique to their domain vs. shared? The plan has structural separation (per-domain blog tables) but no editorial workflow. For a 1-person team this is fine. For 4 domains it will break.

Feature flagging: Not addressed. When a shared component gets a new feature, it appears on all 4 domains simultaneously. This is fine for a portfolio under single ownership — but if you later bring on domain-specific editors or content teams, you'll need feature flags. Low priority now.

Performance budgets: Not addressed. A shared component that loads a 200KB charting library becomes a bundle tax on all 4 apps. The plan mentions "any new dependency affects all apps' bundle sizes" and requires a PR comment — but no hard budget. Consider per-app Lighthouse CI budgets.

Cross-domain user journeys: Not addressed. If a user signs up on LA and then visits Lofts, do they see their favorites? The shared users table suggests yes — but the UX of cross-domain authentication isn't designed. This is a post-launch concern but worth noting.

Monitoring/incident response: Sentry is shared (1 project, domain tags). What happens when shared-package code throws in production? Who gets paged? The plan needs a runbook for "shared package causes errors in 1 of 4 apps."

6. OVERCOMPLEXITY CHECK

Composition manifest + 3-tier system: Justified. Without it, you have no structural guard against doorway pages. The alternative (manual discipline) doesn't scale.

Build-time validator: Justified but could be simpler. Three levels (hard fail, warning, human override) is fine. But don't build this until you actually have 2+ apps in the monorepo (Phase 3+). Building it in Phase 4 as planned is correct.

Token pipeline (TS → CSS generation): Slightly over-engineered for 4 domains. Both apps already use CSS custom properties in globals.css. The generation script adds a layer of indirection that doesn't provide proportional value until you have 6+ brands or non-developer brand authors. However — the satisfies BrandTokens TypeScript contract IS valuable. It catches missing tokens at compile time. Keep the contract, simplify the generation. You could just maintain the CSS files directly and use the TS schema as a validation-only check.

createDomainClient factory with RLS: Justified. The alternative (raw Supabase queries with manual domain_visibility filters scattered through app code) is a guaranteed source of bugs. Centralizing this is correct. BUT — the RLS via set_domain_context + session variable is complex. Since you're using the service role key for ISR (which bypasses RLS), the RLS is a safety net for anon/authenticated queries only. Document this dual-path clearly.

ESLint rule blocking direct @luxurydomains/ui imports: Smart and cheap. Keep it.

7. BARREL FILE PATTERN

Will it work? Yes, with known limitations.

The barrel file pattern (export * from '@luxurydomains/ui'; export { Hero } from '../components/Hero';) works because:

  • TypeScript/Turbopack resolve named exports deterministically (later export wins for conflicts)
  • Tailwind v4's @source directive handles CSS scanning across package boundaries
  • React Server Components don't care about re-exports — they follow the module graph

Known gotchas:

  • Tree-shaking at scale. export * re-exports everything from the UI package even if the app only uses 20 components. Turbopack handles this fine in development (no bundling). In production builds, Next.js uses SWC which does tree-shake — but you should verify bundle sizes after Phase 2.
  • Circular dependency risk. If a shared UI component imports from @luxurydomains/data, and an app's override component also imports from @luxurydomains/data, and the data package somehow references UI types — you get circular resolution. The plan's package boundaries (shared = zero React, UI = zero Supabase, data = zero React) prevent this. Enforce these boundaries with ESLint no-restricted-imports at the package level, not just the app level.
  • @source path in Tailwind. The plan shows @source "../../../packages/ui/src" in globals.css. This works but is fragile — if you reorganize the directory structure, all apps' CSS breaks. Consider using a Tailwind @source alias configured in postcss.config.ts.
  • React Server Components + barrel files. If the barrel file re-exports a client component and a server page imports it, Next.js handles the boundary correctly. BUT: if you add "use client" to the barrel file itself (which some developers do by mistake), it forces all re-exported components to be client components. Add a lint rule: barrel files must never have "use client" directive.

8. DATABASE DESIGN

Verdict: Right hybrid, with one gap.

Shared tables with domain_visibility text[] + GIN index for listings/buildings/neighborhoods is correct. Per-domain tables for blog/editorial/leads is correct. The createDomainClient factory baking in the domain filter is the right abstraction.

The gap: The domain_visibility column defaults to all domains. This means new data automatically appears on every domain. For listings, this might be fine (more coverage). For neighborhoods and cities, it's problematic — you probably don't want luxuryapartments.com/neighborhoods/williamsburg and lofts.com/neighborhoods/williamsburg to both index with the same neighborhood description. The plan should specify which tables default to all-domain visibility vs. which default to the inserting domain only.

Recommended:

  • apartments, buildings: default all domains (coverage maximizer)
  • neighborhoods, cities: default to inserting domain only (content differentiation)
  • states: default all domains (structural, low SEO risk)

Migration concern: The plan calls for a single supabase/ at the monorepo root. This is correct. But the "append-only, timestamp-ordered" convention needs enforcement. Supabase CLI's db diff generates migration files — if two developers create migrations simultaneously with close timestamps, ordering conflicts occur. For a 1-person team, this is fine. For a team, use Supabase's branching feature.

9. CONTENT STRATEGY ENFORCEMENT

Verdict: Structural separation is necessary but not sufficient.

The plan separates blog tables per domain (la_blog_posts, ll_blog_posts). But nothing prevents:

  • Copying a blog post's markdown from LA's table to Lofts' table with s/luxury apartments/lofts/g
  • Reusing the same guide content across domains with minor rewording
  • Publishing the same "Boston Rental Market Guide" on all 4 domains with different CSS

What would actually prevent this:

  • Content fingerprinting in CI. Hash the text content of indexed pages across domains. If similarity > 70% for equivalent routes, warn. This is computationally cheap (simhash/minhash) and catches copy-paste.
  • Editorial calendar per domain. Not a code solution — an operational one. Each domain needs a distinct content strategy: LA covers luxury market segment, Lofts covers industrial/loft conversions, D3 covers whatever its niche is. Content can reference the same data (listings, neighborhoods) but the angle must differ.
  • The composition manifest validator already handles the structural layer. For the content layer, add a text-similarity check as a Level 2 warning (doesn't block builds, but reports in CI).

This isn't in the plan and it should be, even as a future Phase 4 item.
