8. Barrel File Override Pattern

It works, but with documented gotchas:

  • Turbopack + barrel files: Turbopack (Next.js 15+) has optimizePackageImports that should handle barrel files efficiently. But this only works for node_modules by default — workspace packages may not get this optimization. Test explicitly.
  • RSC + re-exports: A barrel file that re-exports a "use client" component from a server context will work — Next.js handles the client/server boundary correctly. However, the barrel file itself becomes a client/server boundary. If you have export { ServerComponent } from './ServerComponent' and export { ClientComponent } from './ClientComponent' in the same barrel, the entire barrel may get treated as a client module if any consumer uses it in a client context. Use separate entry points: @/lib/ui/server and @/lib/ui/client.
  • Tree-shaking: export * from '@luxurydomains/ui' is hostile to tree-shaking. Every component in the package is included in the barrel's module graph. With 60 components, this adds real bundle overhead. Use named re-exports: export { Button, Card, Hero } from '@luxurydomains/ui'.
