6. Do `data-factual` / `data-editorial` HTML attributes align with the engine’s provenance taxonomy?

Short answer

No. They are related but not equivalent. This distinction must be made explicit.

Current meanings

#### Monorepo HTML attributes

  • data-factual: strip from similarity comparison
  • data-editorial: include/prefer for similarity comparison

This is a rendering/extraction classification.

#### Engine provenance taxonomy

  • DETERMINISTIC
  • INFERENTIAL
  • SPECULATIVE
  • UNKNOWN

This is a claim epistemology classification.

Why they don’t map 1:1

Examples:

  • A paragraph saying “Our analysis suggests management responsiveness is above average” is editorial in HTML extraction terms, but inferential in provenance terms.
  • A bulleted score breakdown could be factual in page-layout terms, but contain a mix of deterministic and inferential claims.
  • A “Building Verdict” section is editorial, but may include deterministic facts plus speculative opinion.

So:

  • factual/editorial ≠ deterministic/inferential/speculative

Should they map?

Yes, but as orthogonal dimensions, not a single axis.

A better model:

  • data-content-type="factual|editorial|mixed"
  • data-provenance="deterministic|inferential|speculative|mixed"

Or at least internally in content JSON:

  • blocks carry both presentation_role and provenance_class

Then the renderer can emit attributes accordingly.

Why this matters

Without mapping:

  • similarity tool may overcount or undercount risky text
  • provenance auditing lives only in generation layer, disappears at render layer
  • no way to surface provenance badges in UI if desired

Recommendation

Define block schema like:


type ContentBlock = {
  kind: 'paragraph' | 'score' | 'quote' | 'list';
  role: 'editorial' | 'factual' | 'mixed';
  provenance: 'deterministic' | 'inferential' | 'speculative' | 'mixed';
  text: string;
}

Renderers can then emit:

  • data-editorial
  • data-factual
  • plus optional provenance annotations

Verdict

They align conceptually but are not the same concept. Current docs blur them slightly.
