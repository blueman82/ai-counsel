Question 2: Two-table design

Question 3: Is B-level review sufficient for Tier 2 defense?

It's sufficient for the stated goal, but the framing should be more honest about what it actually buys.

B-level review (5-7 min scan + one field note) provides:

  • Audit trail: A human saw this page before it published. Reviewer ID, timestamp, duration logged. This satisfies Google's Quality Rater "evidence of editorial oversight" signal.
  • One unique human data point: The field note (e.g., "Called leasing office — upper floors quieter") is a genuinely human contribution that no AI could generate. This is the real differentiator.
  • Catch gross errors: A 5-minute read catches hallucinated amenities, wrong neighborhood descriptions, obviously wrong verdicts.

What B-level does NOT provide:

  • Claim verification. The reviewer doesn't check data points against sources.
  • Prose quality editing. Weak sections stay weak.
  • Detection resistance. One inserted sentence changes the statistical profile less than a restructured paragraph would.

For YMYL housing content specifically: B-level is adequate for the "curation" framing (LA curates and reviews buildings) but would be inadequate if the content made specific financial claims. Recommendation: Add a minimum review duration guard — reviews completed in under 2 minutes trigger a "fast review" warning and don't count toward the approval metrics. This prevents rubber-stamping from corrupting the Forge feedback loop.

Question 4: Field notes design

The concept is right. The execution needs one fix.

Separate field with insertion point is architecturally clean. It preserves provenance, enables Forge analysis, and avoids the messy problem of inline editing attribution.

The integration problem: How does a field note with insertion point "within noise section" actually get rendered in the published page? The design describes WHERE to insert but not HOW. The content engine generates prose sections. A field note like "Called leasing office — upper floors quieter" inserted "within noise" needs a rendering template that weaves the human observation into the AI-generated noise section naturally.

If the rendering is naive (append human note after AI paragraph), the seam will be visible — different register, different specificity, different sentence structure. If the rendering is sophisticated (AI rewrites the section incorporating the note), it defeats the provenance separation.

Recommendation: Define the rendering strategy explicitly:

I'd recommend Option A for launch. It's simpler, more honest, and the styled callout is itself an E-E-A-T signal.

Yes, I think the two-table design is directionally correct and better than combining them.

Why separation is good

The queue and the content object are different concerns:

  • editorial_content: canonical content/version/publication state
  • editorial_review_queue: workflow event / review process state

That separation is healthy because:

  • one content item can have multiple review attempts over time
  • Forge requeues published content without replacing live content immediately
  • workflow metadata is noisy and process-oriented
  • content tables remain domain-specific while review process can be shared

This is especially justified by the stated use case: shared queue, per-domain content tables, version replacement, and historical review tracking.

But the current design has one weakness

The queue currently has:

  • editorial_content_id
  • while content also has review_queue_id

That creates a circular coupling and likely unnecessary confusion.

I would recommend:

  • editorial_review_queue references a specific content version row
  • editorial_content should not need review_queue_id unless only one active queue row is ever allowed
  • better model: content version is primary, queue items reference versions
  • if you want “current queue item” on content, derive it via query or add current_review_queue_id explicitly with clear semantics

Better structure

Use:

  • editorial_content_versions or treat current per-domain editorial row as versioned content
  • editorial_review_queue.content_version_id
  • optional parent_queue_id or supersedes_queue_id for revision lineage

So overall:

  • separation: correct
  • current bidirectional linkage: probably too messy
  • combine tables? no
  • refine relational model? yes
